AI 에이전트(Agent)를 위한 LLM을 고를 때는 일반적인 챗봇을 고를 때와는 접근 방식 자체가 완전히 달라야 합니다. 챗봇은 "말을 얼마나 자연스럽게 잘하느냐"가 중요하지만, 에이전트는 **"복잡한 계획을 스스로 짜고, 내가 준 도구(API/코드)를 정확한 타이밍에 엉뚱하지 않게 실행할 수 있느냐"**가 핵심이기 때문입니다. 말 잘하는 우등생보다는 일 잘하는 일잘러를 뽑아야 하는 셈이죠.

### 🛠️ 에이전트용 LLM 선정 기준 ###

#### 1. 함수 호출 및 도구 사용 성공률 (Function Calling / Tool Use) ####
에이전트의 손발이 되는 것이 바로 '도구(Tool)'입니다. LLM이 사용자의 질문을 보고 "지금은 계산기 API를 써야겠군" 하고 판단한 뒤, 개발자가 정해둔 JSON 규격에 맞춰 정확하게 값을 채워 넣는 능력이 핵심입니다.
인자(Arguments) 하나만 철자가 틀려도 에이전트 전체가 뻗어버리므로, 버클리 함수 호출 리더보드(BFCL) 같은 벤치마크 점수가 높은 모델을 골라야 합니다.

#### 2. 추론 및 스스로 오류를 고치는 능력 (Reasoning & Self-Correction) ####
에이전트는 한 번에 일을 끝내지 못하면 여러 단계를 거쳐 해결해야 합니다(ReAct 방식 등). 일을 하다가 에러 메시지를 만났을 때, 당황하지 않고 "아, 이 방법이 틀렸으니 다른 API를 호출해야겠구나" 하고 스스로 궤도를 수정하는 능력이 필수적입니다. 이를 확인하려면 소프트웨어 문제를 직접 해결하는 능력을 측정하는 SWE-bench 점수를 보는 것이 가장 정확합니다.

#### 3. 첫 토큰 출력 속도 및 처리량 (Latency & TTFT) ####
에이전트는 사용자가 질문 하나를 던지면, 백그라운드에서 자기들끼리 생각하고 도구를 부르고 결과를 확인하느라 LLM을 3번, 5번씩 연속으로 호출(Multi-turn loop)하곤 합니다.
한 번 호출할 때마다 멈칫거리며 굼뜨게 행동하면 사용자는 1분 넘게 화면만 바라봐야 합니다. 첫 토큰이 나오는 속도(TTFT)가 극도로 빠른 모델이 에이전트에 유리합니다. 

#### 4. 컨텍스트 윈도우와 '바늘 찾기' 능력 (Context Window & NIAH) ####
에이전트가 돌아가기 시작하면 과거의 대화 기록, 프롬프트, 도구가 뱉어낸 수많은 로그 데이터가 눈덩이처럼 불어나 LLM 입력창으로 들어갑니다.
단순히 수백 KB의 텍스트를 삼킬 수 있는 것뿐만 아니라, 그 방대한 데이터 한가운데 숨겨진 단 하나의 중요한 정보(바늘)를 정확하게 끄집어내는 능력(Needle in a Haystack)이 보장되어야 에이전트가 과거의 맥락을 까먹지 않습니다.

#### 5. 토큰당 비용 (Cost Efficiency) ####
앞서 말씀드렸듯 에이전트는 백그라운드 루프를 엄청나게 돌립니다. 챗봇이 한 번에 1,000토큰을 쓴다면, 에이전트는 뒤에서 혼자 20,000~30,000토큰을 순식간에 태워버리기도 합니다.
따라서 무조건 가장 똑똑하고 비싼 최고 플래그십 모델만 고집하면 영수증을 보고 기절할 수 있습니다. 메인 추론용과 자질구레한 서브 도구 처리용 모델을 나누거나, 가성비가 극대화된 모델을 섞어 써야 합니다.

## 도메인별 선택 ##
에이전트는 단순히 "답변을 주는 것"을 넘어, **'생각(Reason) ➡️ 도구 실행(Act) ➡️ 결과 확인(Observe)'**이라는 자율적인 루프를 돌며 스스로 작업을 완수합니다. 각 영역에서 에이전트가 실제로 어떻게 일하는지 알기 쉽게 정리해 드릴게요.

### 1. 📊 데이터 분석 (Data Analysis) ###
에이전트의 데이터 분석은 사람이 주피터 노트북(Jupyter Notebook)을 열고 코딩하는 과정과 완전히 똑같이 움직입니다. 에어전트가 일하는 방식은 다음과 같읍니다.
* 유저가 CSV나 엑셀 파일을 업로드하며 "이 데이터에서 올해 매출 트렌드를 분석해 줘"라고 명령합니다.
* 에이전트는 내부 파이썬 환경에서 데이터를 먼저 열어보고(EDA), 결측치나 오류가 있는지 스캔합니다.
* 스스로 파이썬 코드(Pandas, Matplotlib 등)를 작성하여 그래프를 그리고 통계 모델을 돌립니다.
* 코드를 실행하다가 에러가 나면, 에러 메시지를 보고 스스로 코드를 수정(Self-Correction)하여 다시 실행합니다.
* 최종 시각화 차트와 함께 핵심 인사이트 리포트를 마크다운 문서로 작성해 유저에게 제출합니다.
*	핵심 무기: 격리된 코드 실행 환경(샌드박스 도커 등), SQL 생성 및 자동 쿼리 능력(Vanna AI 등).

### 2. 💻 자동 코딩 (Automated Coding) ###
2026년 현재의 코딩 에이전트는 한 줄씩 코드를 추천해 주는 '어시스턴트(Copilot)' 단계를 넘어, 스스로 저장소(Repository) 전체를 이해하고 기능을 구현하는 '자율형 엔지니어'로 진화했습니다.
에어전트가 일하는 방식은 다음과 같읍니다.
* 유저가 "로그인 페이지에 '비밀번호 찾기' 버튼을 추가하고 메일 발송 기능 연동해 줘"라고 명령합니다.
*	에이전트는 소스코드 트리 전체를 탐색하며 수정해야 할 파일들(Frontend, Backend, Config)을 찾아냅니다.
*	직접 파일들을 수정하거나 새 코드를 작성합니다.
*	수정이 끝나면 터미널에서 자동으로 테스트 스크립트(npm test, pytest 등)나 빌드 명령어를 실행합니다.
*	테스트가 실패하면 디버깅 모드로 들어가 코드를 다시 고치고, 모든 테스트가 통과(Green)하면 완료 보고를 하거나 Git 커밋/PR을 올립니다.
* 대표적인 도구: Claude Code(터미널 특화), Cursor(IDE 연동), OpenCode, Devin 등.

### 3. 🌐 웹 조사 (Web Research / Deep Research) ###
일반적인 AI 검색이 한 번 검색하고 요약해 주는 단발성 작업이라면, 에이전트의 웹 조사는 "내가 원하는 정보를 찾을 때까지 끝까지 파고드는 딥 리서치(Deep Research)"입니다.
에어전트가 일하는 방식은 다음과 같읍니다.
* 유저가 "2026년 상반기 출시된 글로벌 AI 코딩 에이전트들의 가격과 기능 장단점을 비교 리포트로 만들어 줘"라고 명령합니다.
* 에이전트는 이 거대한 목표를 여러 개의 서브 질문으로 쪼갭니다. (예: 1. Cursor 최신 가격 조사, 2. Claude Code 비용 구조 확인 등)
* 웹 브라우저 도구(Firecrawl, 플레이라이트 등)를 이용해 수십 개의 해외 테크 블로그, 공식 문서, 커뮤니티(Reddit)를 동시다발적으로 크롤링합니다.
*	수집한 데이터 중 광고나 쓸모없는 정보는 걸러내고, 정보가 부족하다 싶으면 스스로 검색어를 바꿔가며 추가 조사(교차 검증)를 수행합니다.
*	완벽하게 팩트 체크가 완료된 방대한 양의 데이터를 표와 카테고리로 깔끔하게 정제하여 보고서를 완성합니다.
*	핵심 무기: 자율 브라우징(웹 스크래핑/크롤링), 정보 신뢰도 필터링 알고리즘.

### 🚀 에이전트 생태계의 공통 분모: MCP (Model Context Protocol) ###
과거에는 데이터 분석용 에이전트, 코딩용 에이전트를 따로 만들어야 했지만, 최근에는 MCP(Model Context Protocol)라는 오픈 표준 덕분에 하나의 에이전트가 이 모든 것을 다 잘할 수 있게 되었습니다. 헤르메스나 Goose 같은 에이전트에 '파이썬 실행 MCP 플러그인', '웹 브라우징 MCP 플러그인'만 꽂아주면 상황에 따라 알아서 변신하며 일을 처리합니다.
